# Documentation on the tabular-to-text pipeline in cruijff_kit

## Input files

These files are generated by the user + Claude in the `create-tabular-schema` and the `design-experiment` skills. With identical `schema.yaml` and `experiment_summary.yaml` fields, the output should be deterministic, but this will not apply if the user opts for the `llm_narrative` or LLM-generated Jinja2 templating (since that will inject variation from the LLM generating the prompting style). 

### schema.yaml

Reusable across experiments; tells the relevant columns, mapping for categorical values, how values should be parsed, and relevant information for perturbations

**Generation:** User can create from scratch (not sure I'd recommend) or Claude can infer a "skeleton" template in convert-tabular-to-text (extracts the categorical metadata from the original file for the value mapping) that the user then reviews and can edit using the `create-tabular-schema` skill. 

#### `columns` schema

- **COLUMN_NAME** — name in the raw tabular dataset
    - **Basic data**
        - `display_name` — human-readable text name (i.e., COW = "class of worker" in ACS)
        - `type` — categorical, numeric, etc.
        - `unit` — textual representation of numeric units for text generation (i.e., "years old" for age)
    - **Perturbation specifications**
        - `synonyms` — list of synonyms for the column name (i.e., "years of age" for "age); randomly-sampled by synonyms.py, so must also include the original display name if the user wants it as an option in the prompts
        - `shorthand_map` — for categorical variables, provides a mapping of potential values to alternative textual representations (i.e., "New York": "NY")
        - `restatements` — alternative semantic ways to express the column to row mapping (i.e., "This person's age is {value}" as default; restatement could be "This person is {value} years old")
    - **value_map**
        - If the tabular values are encoded, this provides the numeric code to the textual representation, akin to a dataset's codebook (i.e., in ACS, SCHL = 18 would be mapped to "Some college, but less than 1 year" for textual representation)

### experiment_summary.yaml

Experiment-specific `data_generation` block; requires a schema.yaml to exist for dataset generation

#### Input prerequisites

- `tool: "tabular_to_text_gen"` indicates that the convert-tabular-to-text skill should be invoked to generate the dataset (not already created, in which case it would be passed through in `data` block)
- `source` — local tabular dataset file (csv, parquet, dta)
- `schema` — the path to the schema file for this source tabular dataset

#### Base prompt components (pre-perturbation/formatting)

- `target` — the column that is to be predicted for this experiment
    - For binary prediction tasks, specify a `threshold` for continuous variables from which the output class should be inferred
- `context` — dataset-level context that applies to each entry.
    - `context_placement`
        - `preamble`: appends the context text to the front of each row-generated prompt (makes sense for base models that do not support system prompts; original usage in folktexts paper)
        - `system_prompt`: for instruction-tuned models, can specify to add as system prompt rather than appended onto each row prompt
- `question` — question that is appended to the end of each generated row prompt
- `emit_source_parquet`: when `true`, will also release a parquet sidecar file with the subsampled rows used to create the respective train/test json files. 
    - `emit_source_parquet_condition`: the condition name (in the case that there are multiple evaluation datasets) that we want the parquet data to match; for example, when changing data representation formats, the underlying tabular rows will be the same across the json files. 

#### Fine-tuning specific arguments

- `split_ratio`: train split for fine-tuning (0.8 = 80% of dataset used for training)
- `validation_ratio`: validation split (0.1 = 10% of dataset used for validation; remaining 10% for evaluation)
- `seed`: when specified, ensures reproducibility because of subsampling and test/train split + random selection of perturbations (i.e., synonyms, reordering)
- `subsampling_ratio`: when 1, will include the whole dataset; otherwise, will do a random sample of that ratio of rows

#### Textual representation specifications

- `conditions` — list of datasets to be generated from the information above that can vary across three axes = features, template, and a list of perturbations
    - `features` — corresponds to the names of the columns in the tabular dataset that should be included in that dataset; must be a subset of the columns specified in the schema file
    - `template`
        - **Dictionary** — bulleted list of column display_name to textual value mappings ("- The age is: 51 years old.")
        - **Narrative** — "English" (full sentence) representation of the data
            - Utilizes Jinja2 templating (.j2 files)
                - Default template that says "The respondent's age is 51 years. Their sex is female..." and so on
                - Or, can have the generate-jinja-template subagent have Claude come up with a custom Jinja2 template for the particular dataset.
        - **llm_narrative**
            - If you have an Anthropic API key, can create custom prompts for every row in the dataset (rather than templating that has the same behavior for every row).
                - THIS IS UNTESTED RIGHT NOW
        - If `narrative` or `llm_narrative` is chosen, the user will be prompted to provide style guidance (e.g., optimizing for token usage, desired narrative form, etc.) that is then passed to the LLM agent that generates either the Jinja2 template or the custom row prompts.
    - `perturbations` — a list of prompt perturbations supported by cruijff_kit
        - **reorder** — shuffles the ordering of columns across rows
        - **clause_addition** — using the `restatements` defined in the schema file, will insert redundant information into the prompt
        - **shorthand** — replaces full-form values with abbreviated forms defined in the schema (i.e., "New York" to "NY")
        - **synonym** — replaces the display name of a column with synonyms defined in the schema file (i.e., "income" versus "yearly earnings"); will randomly select for rows based on the synonyms list
    - `one_to_many` — will expand each source row into N copies (artificially inflating the size of the training set to possibly increase robustness to data format) 
        - `copies` - N to expand each row to 
        - `perturbation` - how the copies should vary from one another (e.g., `reorder` to have the same features but in a different ordering)

## Skills/Agent Workflow

### create-tabular-schema skill

Helper skill (akin to create-inspect-task): 7 step workflow to create the schema YAML file for a particular dataset (e.g., the 2018 ACS)

#### How it works

- **Claude responsible for:**
    - Reporting the columns/types/shapes of the dataset to the user
    - Correctly passing through the user-specified subset of desired columns to the scripts
    - Editing the skeleton schema with user requests
        - Suggesting options for perturbations rephrasing that can be reviewed by the user
    - Validating the YAML syntax and content of the file with the user before finalizing the file
    - Writing the final schema to `{ck_data_dir}/schemas/` in the user's scratch directory
- **User responsible for:**
    - Specifying the desired columns for the schema (AKA any potential target or covariate in the model)
    - Providing a codebook/reference for categorical variables / values not encoded in natural language
    - Adding any potential synonyms, shorthand maps, or restatements (perturbation options)
- **Python responsible for:**
    - Reading the columns of the original dataset (readers.py)
    - Generating the YAML skeleton from the selected columns (schemas.py)

### design-experiment skill

Specifies the dataset generation process for experiments utilizing raw tabular datasets, and helps the user design experiments investigating prompting variants.

#### How it works (param_selection.md step 2)

- **Claude responsible for:**
    - Determining whether the user has the necessary dataset already or whether it needs to be generated
    - Guiding the user through all relevant selections in the `data_generation` block of the `experiment_summary.yaml`
        - Each parameter is described in the design-experiment skill's `references/tabular_to_text_gen.md`
    - If varying datasets across runs (e.g., investigating consequences of a perturbation), adding dataset paths to the `runs` block of the `experiment_summary.yaml`
- **User responsible for:**
    - Providing the raw dataset and a schema.yaml file (created by the create-tabular-schema skill)
    - Specifying what they want to vary in the experiment (e.g, feature set, perturbations) and what datasets apply for training versus evaluation

### convert-tabular-to-text skill

Using a dataset schema, generates text-based .json training/evaluation datasets from raw tabular forms for a particular experiment.

#### How it works

- **Claude responsible for:**
    - Spawning the `generate-jinja-template` subagent if the user wants the LLM to come up with the natural language template, passing through relevant context (`style_guidance`)
    - Determining which combinations of datasets to generate based on the `conditions` table from the `experiment_summary.yaml`
        - Naming the files based on the convention of the condition_train/test_seed.json
    - Invoking the convert.py script with all relevant arguments from the experiment_summary.yaml (step 6)
    - Validating the output (.json formatting, sidecar metadata file, prompting behavior, file splits, etc.) — see validation.md
- **User responsible for:**
    - If LLM-generated, reviewing and approving the created .j2 template from the subagent
- **Scripts responsible for:** the actual .json dataset creation. See the tabular_to_text_gen module documentation below for a more extensive description

#### generate-jinja-template subagent

- Generates a Jinja2 template with the features of the tabular dataset using any style recommendations from the user
    - Exists as a subagent with its own context window, so it does not interact with the user. Rather, it will generate the first pass, then the main agent will ask the user to validate the output + make any adjustments.
- Outputs the generated template in `{ck_data_dir}/templates/` for the user to reuse on same dataset in future

## The text-gen Python Module

**schema.py** — creates a skeleton schema that the user can edit based on their tabular dataset

**Entry point for module is convert.py** and is the only script directly invoked by Claude during convert-tabular-to-text
- Facilitates the pipeline described in ARCHITECTURE.md
- Responsible for the subsampling (seeded train/val/test split) of the original tabular dataset

**readers.py**: Reads the dataset (anything supported by pandas; tested on csv)

**features.py**: Pulls the features specified by the user in the experiment_summary.yaml (in case they want a subset of those in the schema)

**templates.py** (interface): creates a list of Segments (class representing statements for each feature in a row)
- **Dictionary**: parses the "display name: value" based on schema.yaml
- **Narrative**: utilizes the Jinja2 template, the schema.yaml, and the pandas representation of the tabular data to generate the prompts
- **llm_narrative**: calls the Anthropic API with the style guidance from the user in experiment_summary.yaml to generate row-by-row natural language prompts

**perturbations.py** (interface): takes the existing templated list of Segments and applies perturbations
- **clause_addition.py**
- **reorder.py**
- **shorthand.py**
- **synonym.py**
- **engine.py** — chains perturbations together

**segments.py**: Takes the final, templated, perturbed list of segments and writes them into a final textual representation for each row; does not add context appended to the front/back of the prompt

**output.py**: Writes out the final train/validation/test dataset splits in a .json format in the user's `{ck_data_dir}/generated/` folder. Alongside the data, it creates a .meta.json sidecar file with relevant information about (1) configuration for how the data was generated (path to source tabular dataset, selected features, any relevant perturbations), (2) the prompt structure (whether the context is appended to the front of each row or as a system prompt), and (3) metadata (row count, file size). This allows the dataset to be reused across runs and understood by experiment_summary.yaml / analyze-experiment.
